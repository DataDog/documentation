Confirm the SDK is initialized</h3><p><code>DdSdk.Initialize</code> (Pattern 2) and <code>.UseDatadog(...)</code> (Pattern 1) must run before any other Datadog API call. If you initialize from a non-default location (for example, a feature flag–gated startup path), make sure that code runs on every cold launch.</p><p>With verbose logging enabled, the SDK prints a startup message confirming the active site, environment, and feature set.</p><h3 id="2-confirm-tracking-consent-is-">2. Confirm tracking consent is <code>Granted</code></h3><p>If <code>TrackingConsent</code> is <code>Pending</code> or <code>NotGranted</code>, the SDK does not send any data. Update the consent at runtime after the user accepts your privacy dialog:</p><div class="code-snippet-wrapper"><div class="code-filename-wrapper d-flex justify-content-end"></div><div class="code-snippet "><div class="code-button-wrapper position-absolute"><button class="btn text-primary js-copy-button">Copy</button></div><div class="cdoc-code-snippet cdoc-language-csharp"><pre tabindex="0" class="chroma"><code><span class="line"><span class="cl"><span class="n">DdSdk</span><span class="p">.</span><span class="n">SetTrackingConsent</span><span class="p">(</span><span class="n">TrackingConsent</span><span class="p">.</span><span class="n">Granted</span><span class="p">);</span>

Confirm the right Datadog site</h3><p>The default site is <code>Us1</code>. If your organization is on a different site (<code>Us3</code>, <code>Us5</code>, <code>Eu1</code>, <code>Ap1</code>, <code>Ap2</code>, <code>Us1Fed</code>), set <code>Site</code> accordingly on <code>DdSdkConfiguration</code>. The site selector at the top of these documentation pages updates code samples to match.</p><h3 id="4-confirm-the-application-id-and-client-token">4. Confirm the application ID and client token</h3><p>A wrong application ID silently drops events server-side. Double-check the values against the application you created in <a href="https://app.datadoghq.com/rum/application/create"><strong>Digital Experience</strong> &gt; <strong>Add an Application</strong></a>.</p><h2 id="events-appear-but-stack-traces-are-not-symbolicated">Events appear, but stack traces are not symbolicated</h2><p>For native iOS and Android crashes, dSYMs (iOS) and <code>mapping.txt</code> (Android) must be uploaded to Datadog. See <a href="/real_user_monitoring/application_monitoring/maui/error_tracking/#get-deobfuscated-stack-traces">Get deobfuscated stack traces</a> for the full pipeline.</p><p>For managed C# stack traces, Portable PDB files must be bundled in the published app. See <a href="/real_user_monitoring/application_monitoring/maui/error_tracking/#bundle-portable-pdb-files-for-c-stack-traces">Bundle Portable PDB files for C# stack traces</a>.</p><h2 id="c-errors-and-native-crashes-appear-twice">C# errors and native crashes appear twice</h2><p>When <code>NativeCrashReportEnabled = true</code>, an unhandled C# exception that crashes the app is reported both as a managed C# error (via <code>AppDomain.UnhandledException</code>) and as a native crash (via the platform crash reporter). This is intentional — the two events share a session and view so you can correlate them. To keep only one, filter with <a href="/real_user_monitoring/application_monitoring/maui/advanced_configuration/#modify-or-drop-rum-events"><code>ErrorEventMapper</code></a>.</p><h2 id="automatic-actions-are-bucketed-under-the-wrong-view">Automatic actions are bucketed under the wrong view</h2><p><code>TapGestureRecognizer.Tapped</code> and <code>SwipeGestureRecognizer.Swiped</code> only fire on completion, so a tap or swipe that triggers a navigation is recorded against the destination view rather than the source. This is a known limitation that affects only <code>View</code>s with explicit gesture recognizers; <code>Button</code> and <code>ImageButton</code> taps are not affected. See <a href="/real_user_monitoring/application_monitoring/maui/advanced_configuration/#known-limitation-gesture-driven-navigation">Known limitation: gesture-driven navigation</a>.</p><h2 id="view-attributes-attach-to-the-wrong-view">View attributes attach to the wrong view</h2><p><code>DdRum.AddViewAttribute</code> runs against the view that is active in the SDK at the time of the call. If you call it from a page constructor or <code>OnAppearing</code>, the SDK has not yet called <code>StartView</code> for the new page, and the attribute lands on the previous view.</p><p>Call <code>AddViewAttribute</code> from <code>OnNavigatedTo</code> instead — by then the SDK has started the destination view. See <a href="/real_user_monitoring/application_monitoring/maui/advanced_configuration/#add-view-attributes">Add view attributes</a>.</p><h2><code>dotnet publish</code> finishes but symbols are not uploaded</h2><p>The MSBuild target skips silently when <code>datadog-ci</code> is missing or <code>DATADOG_API_KEY</code> is unset.
