## Root cause analysis

Every page is mismatched because of a set of **systemic issues** that affect all pages, not one-off per-page bugs. Fixing these ~8 categories of issues should bring most pages close to parity.

### 1. Missing category-level description

**Hugo** renders the OpenAPI tag's `description` field as an introductory paragraph after the `<h1>` category heading. For example, the Authentication page shows a multi-paragraph introduction about API keys, application keys, and site configuration.

**Astro** skips this entirely. `ApiCategory` in `index.ts` parses the tag description but doesn't include it in the type. `[category].astro` renders only `<h1>` and endpoints.

This is a significant amount of missing text on every page.

### 2. Missing "Overview" heading per endpoint

**Hugo** renders `<h3>Overview</h3>` as a section heading before each endpoint's description text.

**Astro** renders the description directly as `<p>` content with no heading.

The word "overview" is missing from virtually every endpoint diff.

### 3. Custom `x-unstable` / `x-deprecated` messages not rendered

**Hugo** passes the full `x-unstable` string from the OpenAPI spec through `markdownify`, producing messages like:
- "This endpoint is in public beta. If you have any feedback, contact Datadog support."
- "This endpoint uses the legacy security findings data model..."

**Astro** stores `unstable` as a boolean and uses hardcoded generic messages ("This endpoint is unstable and may change without notice."). The actual spec message is discarded in `endpoints.ts` at line 461: `const unstable: boolean = !!operation['x-unstable']`.

This causes widespread "Missing/Extra" word differences on every unstable or deprecated endpoint.

### 4. Missing OAuth scopes display

**Hugo** renders OAuth scopes (from the `security.AuthZ` block) in the endpoint content.

**Astro** extracts `oauthScopes` in `endpoints.ts` but **never renders them** — `ApiEndpoint.astro` only renders `permissions`, not `oauthScopes`. This causes "Missing: oauth, apps, require..." in the diff for many endpoints.

### 5. Full regional URLs vs bare path

**Hugo** renders per-region full URLs as hidden `<span>` elements:
```html
<span class="api-url-pattern d-none" data-region="us">https://api.datadoghq.com/api/v1/validate</span>
```

**Astro** renders only the bare path: `<code>/api/v1/validate</code>`

The QA script strips `.d-none` elements from Hugo, so these hidden URLs should already be excluded. However, the bare path in Astro (e.g., `/api/v1/validate`) is extra text that Hugo doesn't have in its visible content. This accounts for "Extra" words like full URL paths appearing in every Astro endpoint diff.

**Note:** This is partially a QA script issue — the `removeNoise` function strips `.d-none` but Astro's bare path `<code>` has no equivalent noise-removal rule. We need to decide whether to match Hugo's URL display or adjust the QA script to normalize URL paths.

### 6. Missing version tabs / version labels

**Hugo** wraps endpoints in v1/v2 version tabs when both exist. The version label and tab text appear in the extracted content.

**Astro** has a `version` field on `EndpointData` but renders no version indicator. This is a missing feature but has lower impact on text similarity since the tab labels are short.

### 7. 10 Hugo-only pages not generated by Astro

These pages exist in Hugo but are not generated by Astro: `cases`, `cases-projects`, `ci-visibility`, `cloud-workload-security`, `rate-limits`, `scopes`, `service-scorecards`, `team-connections`, `tracing`, `using-the-api`.

Some of these (like `rate-limits`, `scopes`, `using-the-api`) may be static informational pages that don't come from the OpenAPI spec. Others may be categories whose tag names or slugs don't match between Hugo and Astro.

### 8. SchemaTable text concatenation

The Astro `SchemaTable` renders field names, types, and descriptions in separate elements with CSS-only spacing. When cheerio extracts text, adjacent elements without textual whitespace produce concatenated strings like `"parametersnametypedescription..."`.

Similarly, the `DEPRECATED` label in SchemaTable is a `<strong>` followed by a `<span>` with no text separator, producing `"DEPRECATEDDescription text"`.

This artificially inflates the "Extra" word count in Astro and lowers similarity scores, making the comparison noisier than it should be.

## Claude's plan

### Fix 1: Add category-level description to pages

**Files:** `src/data/api/index.ts`, `src/pages/api/latest/[category].astro`

1. Add a `description` field to `ApiCategory` interface (it's already parsed by `TagSchema`)
2. Propagate the tag description into `ApiCategory` in `getApiCategories()`
3. Render the description as markdown HTML after `<h1>` in `[category].astro`

### Fix 2: Add "Overview" heading before endpoint descriptions

**Files:** `src/components/ApiEndpoint.astro`

1. Add `<h3>Overview</h3>` before the `<slot />` that renders the endpoint description
2. Only render the heading when description content exists (check `ep.description`)

### Fix 3: Pass through custom `x-unstable` messages

**Files:** `src/data/api/endpoints.ts`, `src/components/ApiEndpoint.astro`, `src/components/ApiStatusAlert.astro`

1. In `endpoints.ts`, change `unstable` from `boolean` to `string | boolean`:
   - Store the raw `x-unstable` string value (which is the custom message) instead of coercing to boolean
   - Keep it as a truthy/falsy check for the "is it unstable?" question, but preserve the message text
2. Add an `unstableMessage` field to `EndpointData` that holds the rendered markdown of the `x-unstable` string
3. In `ApiEndpoint.astro`, pass the custom message to `ApiStatusAlert` via the existing `message` prop
4. `ApiStatusAlert` already supports a `message` prop and falls back to defaults — no changes needed there

### Fix 4: Render OAuth scopes

**Files:** `src/components/ApiEndpoint.astro`

1. Add an OAuth scopes display section after the permissions section:
   ```astro
   {ep.oauthScopes && ep.oauthScopes.length > 0 && (
     <div class={styles.permissions}>
       <strong>OAuth Apps: </strong>
       <span>This endpoint requires the <code>{ep.oauthScopes.join(', ')}</code> scope(s).</span>
     </div>
   )}
   ```
2. Match the Hugo phrasing for OAuth scopes (check Hugo's template for exact wording)

### Fix 5: Display full regional URLs (or normalize in QA)

**Files:** `src/components/ApiEndpoint.astro` (and possibly `src/data/api/endpoints.ts`)

Two approaches — implement whichever matches Hugo's user-visible behavior:

**Option A — Match Hugo's display:** Add region-aware full URLs to the endpoint display, hidden by default, shown when a region is selected. This requires:
- Extracting region/server data from the spec (the `regions.ts` module was planned but never implemented)
- Rendering hidden `<span>` elements per region, similar to Hugo

**Option B — Adjust QA script:** Add the bare API path `<code>` to the noise-removal list in `extract.ts` for Astro, since Hugo also hides its URL elements. This is simpler but means Astro will still differ from Hugo in the browser.

**Recommendation:** Option A is the correct long-term fix. Option B is a quick workaround if Option A is deferred.

### Fix 6: Generate the 10 missing pages

**Files:** `src/data/api/index.ts`, potentially new static pages

1. Investigate each missing page:
   - Check if the tag exists in the OpenAPI specs with a different name/slug
   - Check if the page is a static informational page (not spec-driven)
2. For spec-driven pages: fix the slug mapping in `index.ts`
3. For static pages (`rate-limits`, `scopes`, `using-the-api`): create equivalent static content pages in Astro, either as `.astro` pages or content files

### Fix 7: Add whitespace to SchemaTable text nodes

**Files:** `src/components/SchemaTable.tsx`

1. Add a text space between the `DEPRECATED` `<strong>` and the description `<span>`:
   ```tsx
   {field.deprecated && (
     <><strong class={styles.deprecated__label}>DEPRECATED</strong>{' '}</>
   )}
   ```
2. Ensure table header cells and data cells have proper text-node whitespace between them (add space text nodes or use CSS that cheerio can detect)

This won't change the visual appearance (CSS already provides spacing via margins) but will make text extraction produce correctly separated words.

### Fix 8: Add version indicator to endpoints

**Files:** `src/components/ApiEndpoint.astro`

1. When `ep.version` is available, render a small version badge near the endpoint heading (e.g., "v1", "v2")
2. This is lower priority since version labels contribute minimal text to the diff

### Implementation order

1. **Fix 1** (category description) — highest impact, affects every page
2. **Fix 3** (x-unstable messages) — high impact, affects all unstable/beta/deprecated endpoints
3. **Fix 4** (OAuth scopes) — high impact, affects many endpoints
4. **Fix 2** (Overview heading) — medium impact, single missing word per endpoint
5. **Fix 7** (SchemaTable whitespace) — medium impact, improves QA accuracy across all pages

7. **Fix 5** (regional URLs) — lower impact, mostly a QA noise issue
8. **Fix 8** (version indicator) — lowest impact
