## Claude's plan

### Data approach: Zod-validated utility module (not a content collection)

The API spec files are large OpenAPI 3.0 YAML files (v1: ~39k lines, v2: ~118k lines), not individual content documents. A content collection doesn't fit here because:
- Each file contains hundreds of endpoints, not one piece of content
- There's no markdown body to render — just structured data
- We need to slice the data by tag (category) at build time, not load individual files as entries

Instead, I'll create a **data utility module** (`src/data/api/index.ts`) that:
1. Reads the copied YAML spec files at build time using `fs` + a YAML parser (`yaml` npm package)
2. Validates the relevant portions of the data with Zod (tags, paths/operations, `x-menu-order`)
3. Exports typed helper functions like `getApiCategories()` and `getOperationsForTag(tag)`

This keeps things simple and explicit. The Zod schema only validates the fields we actually use (tag names, operation summaries, `x-menu-order`, `operationId`), not the entire OpenAPI spec — which would be enormous and unnecessary for a nav shell.

### Files to copy

The Astro site must be fully self-contained — it should not depend on any files from the Hugo site at build or run time. All required assets (spec files, images, etc.) are copied into the Astro directory tree.

- Copy `data/api/v1/full_spec.yaml` and `data/api/v2/full_spec.yaml` into `astro/src/data/api/`

We skip the translation and code example JSON files for now since we're only building the nav shell.

### New design tokens

The sidebar and nav styling will use existing general-purpose tokens (`--color-bg`, `--color-border`, `--color-brand`, `--color-text-muted`, etc.). The only new token needed is:
- `--sidebar-width` — layout dimension for the nav column width (matching Hugo's 3/12 ≈ 25%)

### Visual fidelity

Match the cosmetic appearance of the Hugo API docs as closely as possible — font sizes, spacing, colors, layout proportions, active states, etc. — without breaking our rules for design tokens or CSS modules. All values should come from tokens (adding new general-purpose tokens if needed), and all component styles should live in CSS modules. Where Hugo uses hard-coded values or Bootstrap utilities, translate them into token-driven equivalents.

### Left nav structure

Looking at the Hugo output, the nav is a two-level hierarchy:

```
Datadog Docs API (header)
├── Overview
├── Action Connection          ← tag/category (links to /api/latest/action-connection/)
│   ├── Get an existing...     ← operation (anchor link #get-an-existing-action-connection)
│   ├── Create a new...
│   └── ...
├── Dashboards
│   ├── Create a new dashboard
│   ├── List dashboards
│   └── ...
└── ... (100+ categories from v1+v2 combined)
```

I'll recreate this with:
- **`ApiSideNav.astro`** — Static component rendering the full nav tree. Categories are `<a>` links to their page; operations are nested `<a>` links with `href` anchors. Current page is highlighted.
- **Sticky positioning** via CSS (`position: sticky; top: 0`)
- **Collapsible categories** — only the active category's operations are expanded (CSS-only, no JS needed for the shell)
- **Alphabetical ordering** of categories, with operations ordered by `x-menu-order`

### Page generation

Use Astro's `getStaticPaths()` in a dynamic route file to generate one page per API category:

- **Route**: `src/pages/api/latest/[category].astro`
- **`getStaticPaths()`** calls the data utility to get all unique tags (merged from v1+v2), and generates a page for each
- **Slug derivation**: Kebab-case the tag name (e.g., "Key Management" → `key-management`), matching Hugo's URL pattern
- **Page content**: Title (`<h1>`) + a placeholder line for each operation: "This will be the page for the {operation summary} endpoint."

### Questions/risks to flag

- **Spec file size**: The v2 spec is 4MB. Parsing it at build time should be fine for static generation, but if build times become a concern, we could cache the parsed result. I'll keep the implementation simple for now.
- **v1 vs v2 merge strategy**: Some tags exist in both v1 and v2 (e.g., "Dashboards"). Hugo shows version tabs per endpoint. For the shell, I'll merge operations under a single category and note which version each comes from. The detailed version-switching UI can come later.
- **Endpoint anchors**: Hugo uses `#slug-of-operation-summary` as anchor IDs. I'll use the same pattern for consistency.
- **The prompt says "page for each API endpoint"**: I'm interpreting this as one page per tag/category (matching Hugo's structure, where e.g. `/api/latest/dashboards/` contains all dashboard endpoints), not one page per individual operation. The Hugo site groups operations by tag on a single page. If you want one page per individual operation instead, let me know.
