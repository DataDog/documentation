## 目標
一般的なシェルユーティリティ、HTTP ユーティリティ、または言語エンジン（Python や PHP）またはウェブサービス（nginx など）によって生成されたシェルを検出します。

## 戦略
Web シェルの攻撃では、攻撃者が悪意のあるファイルを犠牲となるマシンに読み込み実行し、侵害されたシステムにバックドアを作成することがよくあります。攻撃者はさまざまな目的で Web シェルを使用し、侵入やより広範囲への攻撃の始まりを示す場合があります。この検出により、一般的なシェルユーティリティ、HTTP ユーティリティ、またはシェルが一般的なウェブサーバーまたは言語エンジン（たとえばウェブアプリケーションの一部として）によって生成された時にトリガーされます。これは、ウェブサーバーまたは言語エンジンにとって非典型的な動作です。これが予期しない動作である場合は、攻撃者が既存の Web シェルを使用またはインストールしようとしていることを示している可能性があります。

## トリアージと対応
1. ウェブアプリケーションがシェルとユーティリティを実行するための承認された目的があるかどうかを判断します。
2. この動作が予期しないものである場合は、侵害の封じ込めを試みます (攻撃の段階によっては、ワークロードを終了することで実現できる場合があります)。アプリケーションログまたは APM データを調査して、最初の侵害の兆候を探します。侵害されたシステムを調査および修正するために、組織の内部プロセスに従います。
3. エクスプロイトの根本原因を見つけて修復します。
