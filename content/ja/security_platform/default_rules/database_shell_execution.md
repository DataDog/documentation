## 目標
一般的なシェルユーティリティ、HTTP ユーティリティ、またはデータベースプロセスによって生成されたシェルを検出します（MySQL、PostgreSQL、MongoDB など）。

## 戦略
データベースへの攻撃は、通常 I/O サニタイズおよび検証において攻撃者ステートメントやコマンドの実行を見過ごしてしまうことを利用されます。たとえば、攻撃はデータベースのクエリ挿入の形式をとることがあり、Web シェルの確立またはデータ流出により、侵入およびより広範囲への攻撃の始まりを示す場合があります。この検出により、一般的なシェルユーティリティ、HTTP ユーティリティ、またはシェルがデータベースプロセス（MySQL、MongoDB、PostgreSQL）によって生成された時にトリガーされます。これは、データベースにとって非典型的な動作です。これが予期しない動作である場合は、攻撃者がデータベースまたはホストマシンを危険にさらそうとしていることを示している可能性があります。

## トリアージと対応
1. データベースがシェルとユーティリティを実行するための承認された目的があるかどうかを判断します。
2. この動作が予期しないものである場合は、侵害の封じ込めを試みます (攻撃の段階によっては、ワークロードを終了することで実現できる場合があります)。アプリケーションログまたは APM データを調査して、最初の侵害の兆候を探します。侵害されたシステムを調査および修正するために、組織の内部プロセスに従います。
3. エクスプロイトの根本原因を見つけて修復します。
